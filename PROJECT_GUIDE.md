# 🌌 Nebula Chat — Official Project Guide

Welcome to the **Nebula Chat** documentation. This guide provides a comprehensive overview of the project's architecture, features, and code implementation.

---

## 🚀 1. Project Overview
Nebula Chat is a modern, real-time web application designed for seamless communication. It allows users to join a **Global Room** for public broadcasting or engage in **Direct Messaging (DM)** for private conversations.

### Key Capabilities:
- **Instant Messaging**: Powered by WebSockets for sub-second latency.
- **Image Sharing**: Send images directly in the chat (up to 5MB).
- **Persistent History**: All messages are stored in a local SQLite database and retrieved on login.
- **Glassmorphism UI**: A premium, dark-mode design with smooth animations and responsive layouts.
- **Multi-Platform Deployment**: Configured for Railway (Backend), GitHub Pages (Frontend), and Hugging Face Spaces.

## 🧠 4. Deep Dive: Backend (`main.py`)

The backend is built with **FastAPI**. It handles three major responsibilities:

### A. Database Management
- Uses `sqlite3` to store users and messages.
- **Schema**:
    - `users`: Stores unique usernames and their `last_seen` timestamp.
    - `messages`: Stores `sender`, `recipient`, `content`, `type` (text/image), and `timestamp`.
- **Initialization**: `init_db()` ensures tables exist and handles schema migrations (e.g., adding the `type` column).

### B. REST API Endpoints
- `POST /api/login`: Validates the username and records/updates the user in the database.
- `GET /api/users`: Returns a list of all registered users.
- `GET /api/messages/{user1}/{user2}`: Retrieves the last 200 messages between two users (or from the Global Room).

### C. WebSocket Management
The `ConnectionManager` class tracks active connections in a dictionary (`active_connections`).
- **Broadcast**: Sends a message to every connected user (used for Global Room).
- **Personal Message**: Sends a message to a specific user's socket (used for DMs).
- **Persistence**: Every message received via WebSocket is instantly saved to the database before being routed to the recipient.

## 🎨 5. Deep Dive: Frontend (`script.js` & `style.css`)

The frontend is a **Single Page Application (SPA)** written in vanilla JS to minimize overhead.

### A. The WebSocket Client
The `connectWebSocket()` function manages the lifecycle of the connection:
1.  Converts the `RAILWAY_URL` from `https` to `wss`.
2.  Handles `onopen`, `onmessage`, and `onclose`.
3.  **Auto-Reconnect**: If the server goes down, the client attempts to reconnect every 3 seconds.

### B. Message Rendering
- **Security**: Uses `textContent` for text messages to prevent XSS.
- **Multimedia**: Detects `msg.type === 'image'` and renders an `<img>` tag with the Base64 content.
- **Context Aware**: The UI filters incoming messages so you only see the chat you are currently "in."

### C. Image Handling
Images are handled using the `FileReader` API:
1.  User selects an image via `<input type="file">`.
2.  JS converts the file to a **Base64 DataURL**.
3.  The DataURL is sent as a string over the WebSocket with `type: 'image'`.
4.  The backend stores this string, and receivers render it as the `src` of an image.

## ☁️ 7. Deployment Guide

### Backend (Railway)
- Connect your GitHub repo to Railway.
- Railway will automatically detect the `Procfile` and `requirements.txt`.
- Set the `PORT` environment variable if needed.

### Frontend (GitHub Pages)
1.  Configure the repository to serve from the `/docs` folder.
2.  **Crucial**: Update `RAILWAY_URL` in `docs/script.js` to point to your live Railway API.

## 📝 8. Summary of Logic Flow
1.  **User Logs In** → `POST /api/login` → Server saves user → Client connects to WebSocket.
2.  **Select User** → Client fetches history via `GET /api/messages/...`.
3.  **Send Message** → Client sends JSON via WebSocket → Server saves to DB → Server pushes JSON to recipient(s).
4.  **Receive Message** → Client checks if the message belongs to current view → Appends to DOM.
